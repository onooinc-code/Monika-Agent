The Development Cycle

Our feature-driven development process ensures clarity, quality, and traceability. The entire lifecycle, from idea to production, is designed to be systematic and is followed by both human and AI developers.

The Six Phases of Development

Each new feature or significant change goes through a structured, six-phase lifecycle.

Phase 1: Idea & Feature Definition

A new feature begins its life as a formal specification document. This document clearly outlines the feature's objectives and core functionalities, serving as the single source of truth for the entire development team.

  • Artifact: A .txt file (e.g., 15-streaming-responses.txt).
  • Location: Initially created in Monica-Documentation/features/Remaining/.

Phase 2: Planning

Before any code is written, a detailed implementation plan is formulated. This crucial step involves architectural decisions, such as identifying which files will be created or modified, defining new functions and components, and outlining the state management strategy. This prevents costly rework and ensures the new feature integrates cleanly with the existing architecture.

Phase 3: Implementation

The developer (human or AI) executes the plan, writing clean, efficient, and well-documented code that fulfills the requirements laid out in the feature definition. All code must adhere to the project's established coding standards.

Phase 4: Testing

Quality is paramount. A comprehensive QA test plan is generated, covering all aspects of the new feature. This plan includes static code analysis, risk assessment, unit/integration test scenarios (written in Gherkin syntax for clarity), and a step-by-step manual E2E test plan.

  • Artifact: A .md test plan (e.g., 15-streaming-responses.md).
  • Location: Created in Monica-Documentation/testing/.

Phase 5: Completion & Documentation

Upon successful implementation and testing, the feature is considered complete. The original feature definition file is moved from the Remaining to the Completed directory, creating a clear and auditable record of progress. Any relevant architectural or user-facing documentation is updated at this stage.

Phase 6: Production

The feature is merged into the main branch and deployed to production, making it available to users. The cycle then repeats for the next feature in the backlog.
